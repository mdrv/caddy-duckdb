# Caddy x DuckDB Integration — v0.1.0 Plan

> **Status:** Draft — pending approval\
> **Date:** 2026-06-03\
> **Scope:** Caddy HTTP module that embeds DuckDB via CGo, enabling SQL queries defined in Caddyfile (or external `.sql` files) to execute against HTTP requests with parameter binding from GET query strings, POST bodies, headers, and path segments.

---

## 1. Motivation

The original plan (in `claude.md`) covered a full analytics + abuse-detection system with ~2300 lines of architecture. While comprehensive, the primary value proposition is simpler:

**Write SQL in your Caddyfile, bind it to HTTP routes, and Caddy executes it directly against DuckDB.**

This eliminates the need for sidecar services (Bun, Node, etc.) for simple data operations. The canonical motivating use case is **GPS tracking with Colota** — currently requiring a separate Bun server (`/g/colota/colota-server-setup.md`, Option B) — which can be reduced to a single `INSERT` query in a Caddyfile block.

### What this replaces

| Before                                                       | After                                 |
| ------------------------------------------------------------ | ------------------------------------- |
| Caddy reverse_proxy → Bun → SQLite                           | Caddy module → DuckDB directly        |
| ~40-line Bun server + systemd unit + separate process        | One Caddyfile block                   |
| Separate data stores (SQLite for Colota, logs for analytics) | One DuckDB file, queryable from Caddy |

---

## 2. Architecture

```
Client (browser, app, curl)
  │
  ▼
Caddy (with caddy-duckdb module)
  ├── Request logging ──► DuckDB (async batch INSERT, automatic)
  ├── Named query routes ──► DuckDB (sync, user-defined SQL)
  ├── External .sql file routes ──► DuckDB (sync, file-referenced SQL)
  └── Static file serving ──► filesystem (for dashboards, maps, etc.)
```

The module does two things:

1. **Passive logging** — every request through Caddy gets recorded asynchronously (batch writer). This is the analytics baseline.
2. **Active query execution** — user-defined SQL runs on matched routes, reading from or writing to DuckDB based on request parameters.

## 3. Core Feature: Caddyfile-Native SQL

### 3.1 Inline queries

Define SQL directly in the Caddyfile using backtick blocks:

```caddyfile
duckdb {
    db_path /var/lib/caddy/data.db

    query insert_location {
        sql `
            INSERT INTO locations (received, lat, lon, acc, alt, vel, bear, batt, bs, tst)
            VALUES (
                epoch(now()),
                $lat, $lon, $acc, $alt, $vel, $bear, $batt, $bs, $tst
            )
        `
        param lat  { from body, key lat,  type float }
        param lon  { from body, key lon,  type float }
        param acc  { from body, key acc,  type float, default 0 }
        param alt  { from body, key alt,  type float, default null }
        param vel  { from body, key vel,  type float, default null }
        param bear { from body, key bear, type float, default null }
        param batt { from body, key batt, type int,   default null }
        param bs   { from body, key bs,   type int,   default null }
        param tst  { from body, key tst,  type int,   default 0 }
        output {
            status 200
            body ""
        }
    }

    route POST /api/location insert_location
}
```

When Colota POSTs `{"lat":-6.9222,"lon":107.6070,"acc":12,...}` to `/api/location`, the module:

1. Matches the route `POST /api/location`
2. Parses the JSON body, binding `$lat`, `$lon`, etc. from the payload
3. Executes the `INSERT` against DuckDB
4. Returns HTTP 200

No Bun. No systemd service. No separate process.

### 3.2 External SQL files

For longer queries or queries shared across sites, reference an external `.sql` file:

```caddyfile
duckdb {
    db_path /var/lib/caddy/data.db

    query recent_locations {
        sql_file /etc/caddy/queries/recent_locations.sql
        param limit  { from query, key limit,  type int, default 50, cap 500 }
        param since  { from query, key since,  type timestamp }
        output {
            format json
            envelope on
        }
    }

    route GET /api/locations recent_locations
}
```

**`/etc/caddy/queries/recent_locations.sql`:**

```sql
SELECT
    tst,
    lat,
    lon,
    acc,
    batt,
    received
FROM locations
WHERE ($since IS NULL OR tst > $since)
ORDER BY tst DESC
LIMIT $limit
```

### 3.3 Parameter sources

Parameters can come from any part of the HTTP request:

| Source               | Directive          | Example                                |
| -------------------- | ------------------ | -------------------------------------- |
| URL query string     | `from query`       | `?limit=50` → `$limit` = 50            |
| JSON POST body       | `from body`        | `{"lat": -6.9}` → `$lat` = -6.9        |
| Request header       | `from header`      | `Authorization: Bearer xyz` → `$token` |
| URL path segment     | `from path`        | `/api/locations/:id` → `$id`           |
| Environment variable | `from env`         | `COLOTA_TOKEN` → `$token`              |
| Caddy placeholder    | `from placeholder` | `{remote_host}` → `$ip`                |

### 3.4 Parameter types and validation

```caddyfile
param limit {
    from    query       # where to read the value
    key     limit       # key name in that source
    type    int         # coercion: string | int | float | bool | timestamp | duration
    default 50          # fallback when absent
    min     1           # validation: minimum value
    cap     500         # hard ceiling (silently caps, unlike max which rejects)
    pattern ^\d+$       # regex validation (string type only)
}
```

- `min` / `max` → reject the request if out of range (400 error)
- `cap` → silently clamp the value instead of rejecting
- `pattern` → regex match for string params

## 4. Caddyfile Grammar

Complete directive reference:

```
duckdb {
    # ── Database connection ───────────────────────────────
    db_path       <path>              # Path to .db file (required). ":memory:" for ephemeral.
    db_setting    <key> <value>       # DuckDB SET options (repeatable)
    read_only     [on|off]            # Read-only mode
    retention     <duration>          # Auto-delete request logs older than this (default 90d)
    checkpoint    <duration>          # WAL checkpoint interval (default 6h)

    # ── Write pipeline (request logging) ──────────────────
    batch_size      <int>             # Records per flush (default 500)
    flush_interval  <duration>        # Max time between flushes (default 200ms)
    buffer_size     <int>             # Channel capacity (default 8192)
    overflow        [drop|block]      # Behavior when buffer full (default drop)
    exclude_path    <pattern>         # Never log these paths (repeatable)
    exclude_field   <field>           # Omit from request records (repeatable)

    # ── Named query definition (repeatable) ───────────────
    query <name> {
        sql `<SQL>`                   # Inline SQL (backtick-quoted)
        # OR:
        sql_file <path>               # Path to external .sql file

        param <name> {
            from     [query|body|header|path|env|placeholder]
            key      <string>
            type     [string|int|float|bool|timestamp|duration]
            default  <value>
            min      <number>
            max      <number>
            cap      <number>
            pattern  <regex>
        }

        output {
            format       [json|json_array|csv|ndjson]
            envelope     [on|off]              # wrap in {data:[], meta:{}} (default on)
            alias        <column> <as>         # rename column in output (repeatable)
            omit         <column>              # drop column from output (repeatable)
            status       <int>                 # HTTP status code (default 200)
            body         <string>              # literal response body (for INSERTs returning "")
        }

        cache    <duration>           # Cache query results for this long
        timeout  <duration>           # Per-query timeout override (default 30s)
    }

    # ── Route binding (repeatable) ────────────────────────
    route <METHOD> <path> <query_name> {
        require_header <name> <value>   # Auth gate (repeatable)
        rate_limit <n> per <duration>   # Per-route rate limit
    }

    # ── Internal analytics API ────────────────────────────
    api_path      <path>              # Mount point (default "/_duckdb")
    api_token     <token>             # Bearer token for the API
    raw_sql       [on|off]            # Allow POST /sql with arbitrary SQL (default off)
    max_rows      <int>               # Row limit on all queries (default 10000)
    cors_origin   <origin>            # CORS header for browser access

    # ── Parquet partitioning (repeatable table) ───────────
    partitioning {
        table      <name>             # Table to rotate (repeatable)
        interval   [daily|weekly|monthly]  # Rotation frequency (default daily)
        format     [parquet|csv]      # Archive format (default parquet)
        path       <directory>        # Where to store archive files
        filename   <template>         # {table}_{date}.parquet (default)
        keep_days  <int>              # Days to keep in active DB before archiving (default 1)
        compress   [zstd|snappy|gzip|none]  # Compression (default zstd)
        max_age    <duration>         # Delete archives older than this (default: never)
    }
}
```

## 5. Real-World Example: Colota GPS Tracking

This is the primary motivating use case. A complete Caddyfile that replaces the entire Bun + systemd setup from `/g/colota/colota-server-setup.md` Option B:

```caddyfile
{
    order duckdb before reverse_proxy
}

location.example.com {
    duckdb {
        db_path /var/data/colota/locations.db

        db_setting memory_limit 256MB

        batch_size     100
        flush_interval 500ms
        exclude_path   /api/location
        exclude_path   /api/locations
        exclude_path   /favicon.ico

        # ── Parquet archiving (keeps DB small for years of GPS data) ──
        partitioning {
            table      locations
            interval   daily
            format     parquet
            path       /var/data/colota/archive
            filename   {table}_{date}.parquet
            keep_days  7
            compress   zstd
            max_age    365d
        }

        # ── Schema bootstrap ──────────────────────────────
        query bootstrap_locations {
            sql `
                CREATE TABLE IF NOT EXISTS locations (
                    id        INTEGER PRIMARY KEY DEFAULT nextval('loc_seq'),
                    received  DOUBLE   NOT NULL,
                    lat       DOUBLE   NOT NULL,
                    lon       DOUBLE   NOT NULL,
                    acc       DOUBLE,
                    alt       DOUBLE,
                    vel       DOUBLE,
                    bear      DOUBLE,
                    batt      INTEGER,
                    bs        INTEGER,
                    tst       BIGINT
                );
                CREATE SEQUENCE IF NOT EXISTS loc_seq;
                CREATE INDEX IF NOT EXISTS idx_loc_tst ON locations (tst);
            `
            output { status 204 }
        }

        # ── Colota POST ingestion ─────────────────────────
        query insert_location {
            sql `
                INSERT INTO locations (received, lat, lon, acc, alt, vel, bear, batt, bs, tst)
                VALUES (
                    epoch(now()),
                    $lat, $lon,
                    COALESCE($acc, 0),
                    $alt, $vel, $bear,
                    $batt, $bs,
                    COALESCE($tst, 0)
                )
            `
            param lat  { from body, key lat,  type float }
            param lon  { from body, key lon,  type float }
            param acc  { from body, key acc,  type float }
            param alt  { from body, key alt,  type float }
            param vel  { from body, key vel,  type float }
            param bear { from body, key bear, type float }
            param batt { from body, key batt, type int }
            param bs   { from body, key bs,   type int }
            param tst  { from body, key tst,  type int }
            output { status 200, body "" }
        }

        # ── Location history (GET with query params) ──────
        query get_locations {
            sql `
                SELECT tst, lat, lon, acc, batt, received
                FROM locations
                WHERE ($since IS NULL OR tst > $since)
                  AND ($until IS NULL OR tst < $until)
                ORDER BY tst DESC
                LIMIT $limit
            `
            param since { from query, key since, type timestamp }
            param until { from query, key until, type timestamp }
            param limit { from query, key limit, type int, default 100, cap 5000 }
            output { format json, envelope on }
        }

        # ── Last known position ───────────────────────────
        query last_location {
            sql `
                SELECT tst, lat, lon, acc, batt, vel, bear
                FROM locations
                ORDER BY tst DESC
                LIMIT 1
            `
            output { format json, envelope off }
        }

        # ── Daily summary ─────────────────────────────────
        query daily_summary {
            sql `
                SELECT
                    DATE(epoch_ms(tst * 1000)) AS date,
                    COUNT(*)                    AS points,
                    MIN(lat)                    AS min_lat,
                    MAX(lat)                    AS max_lat,
                    MIN(lon)                    AS min_lon,
                    MAX(lon)                    AS max_lon,
                    AVG(batt)                   AS avg_battery
                FROM locations
                WHERE tst > $since
                GROUP BY date
                ORDER BY date DESC
                LIMIT $limit
            `
            param since { from query, key since, type int, default 0 }
            param limit { from query, key limit, type int, default 30, cap 365 }
            output { format json, envelope on }
        }

        # ── Routes ────────────────────────────────────────
        route POST /api/location insert_location {
            require_header Authorization "Bearer {env.COLOTA_TOKEN}"
        }

        route GET /api/locations get_locations {
            require_header Authorization "Bearer {env.COLOTA_TOKEN}"
        }

        route GET /api/locations/last last_location {
            require_header Authorization "Bearer {env.COLOTA_TOKEN}"
        }

        route GET /api/locations/daily daily_summary {
            require_header Authorization "Bearer {env.COLOTA_TOKEN}"
        }
    }

    # Static files for a map dashboard (optional)
    root * /var/www/colota
    file_server
}
```

**What this replaces:**

- `/opt/colota/server.ts` (40 lines of Bun)
- `/etc/systemd/system/colota.service` (systemd unit)
- A running Bun process (~20MB RAM)
- A separate SQLite database

**What you gain:**

- Single process (Caddy + DuckDB)
- Automatic TLS (Caddy's Let's Encrypt)
- Queryable OLAP database (DuckDB > SQLite for analytics)
- HTTP API for location history, last position, daily summaries — for free

## 7. Module Registration

```go
package caddyduckdb

import "github.com/caddyserver/caddy/v2"

func init() {
    caddy.RegisterModule(Middleware{})
}

type Middleware struct {
    DBPath          string            `json:"db_path"`
    BatchSize       int               `json:"batch_size,omitempty"`
    FlushInterval   caddy.Duration    `json:"flush_interval,omitempty"`
    Queries         []QueryDef        `json:"queries,omitempty"`
    Routes          []QueryRoute      `json:"routes,omitempty"`
    ExcludePaths    []string          `json:"exclude_paths,omitempty"`
    ExcludeFields   []string          `json:"exclude_fields,omitempty"`

    db     *sql.DB
    writer *BatchWriter
    reg    *QueryRegistry
    routes map[string]*matchedRoute
}

func (Middleware) CaddyModule() caddy.ModuleInfo {
    return caddy.ModuleInfo{
        ID:  "http.handlers.duckdb",
        New: func() caddy.Module { return new(Middleware) },
    }
}
```

## 8. Request Flow

```
HTTP Request arrives
  │
  ├─ Match a `route` directive? ── YES ──► Resolve params from request
  │                                         │
  │                                         ├─ Auth check (require_header)
  │                                         ├─ Rate limit check
  │                                         ├─ Execute query (or cached result)
  │                                         ├─ Format output
  │                                         └─ Write response
  │
  └─ No route match ──► Pass to next handler (reverse_proxy, file_server, etc.)
                            │
                            └─ After response: async write request record to DuckDB
                               (unless path is in exclude_path)
```

Key behaviors:

- **Route-matched queries** run synchronously — the client waits for the result
- **Request logging** is always async (batch writer) and never blocks the response
- Routes with `require_header` that fail auth return 401 before executing SQL
- Query results are cached per `cache <duration>` — subsequent identical requests serve from memory

## 9. Parameter Binding Detail

### JSON body parsing

When `from body` is used, the module reads the request body once, parses it as JSON, and extracts the value at the given key:

```json
{ "lat": -6.9222, "lon": 107.607, "acc": 12 }
```

```
param lat { from body, key lat, type float }  →  $lat = -6.9222
param lon { from body, key lon, type float }  →  $lon = 107.607
param alt { from body, key alt, type float }  →  $alt = nil (key absent, no default → NULL)
```

Nested keys use dot notation: `key geo.lat` → `{"geo": {"lat": 1.0}}` → `$lat = 1.0`

### Type coercion

| Type        | Go type     | DuckDB type   | Notes                                  |
| ----------- | ----------- | ------------- | -------------------------------------- |
| `string`    | `string`    | `VARCHAR`     | Regex pattern matching via `pattern`   |
| `int`       | `int64`     | `BIGINT`      | `min`/`max` reject; `cap` clamps       |
| `float`     | `float64`   | `DOUBLE`      | `min`/`max` reject; `cap` clamps       |
| `bool`      | `bool`      | `BOOLEAN`     | Accepts `true/false`, `1/0`, `yes/no`  |
| `timestamp` | `time.Time` | `TIMESTAMPTZ` | RFC3339 format: `2024-01-15T10:30:00Z` |
| `duration`  | `string`    | `INTERVAL`    | Go duration syntax: `24h`, `30m`, `7d` |

### NULL handling

- If a param is absent from the request **and** has no `default`, it binds as SQL `NULL`
- If a param has `default null`, it explicitly binds as `NULL`
- Use `COALESCE($param, fallback)` in SQL for clean defaults at the query level

## 10. External SQL Files

### Loading

```caddyfile
query get_locations {
    sql_file /etc/caddy/queries/locations.sql
    param limit { from query, key limit, type int, default 100 }
}
```

The module reads the file at **provision time** (during `caddy reload`), not on every request. This means:

- Syntax errors in the SQL file are caught at config load (fail-fast)
- Changing a query requires `caddy reload` (consistent with Caddy's model)

### File watching (optional enhancement)

A future version may support hot-reloading SQL files without `caddy reload` via `fsnotify`. For v0.1.0, explicit reload is sufficient and safer.

## 11. Parquet Partitioning (Day-Based Rollover)

### Problem

GPS tracking runs for years. A single DuckDB `.db` file grows unboundedly — after months of continuous Colota pings, it becomes hundreds of MB. DuckDB handles large files well, but operational hygiene demands rotation.

### Solution: Active DB + Parquet Archive

```
/var/data/colota/
├── locations.db                           # active DB (today's data + recent)
└── archive/
    ├── locations_2026-06-01.parquet
    ├── locations_2026-06-02.parquet
    └── locations_2026-06-03.parquet
```

**How it works:**

1. All writes go to the active DuckDB `.db` file (fast, transactional, append-optimized)
2. At day boundary (configurable), a background task:
   - Exports yesterday's rows to `archive/<table>_YYYY-MM-DD.parquet` via `COPY TO`
   - Deletes the exported rows from the active DB
   - Runs `CHECKPOINT` to reclaim disk space
3. Queries transparently span both sources using DuckDB's `read_parquet()` + `UNION ALL`

### Caddyfile configuration

```caddyfile
duckdb {
    db_path /var/data/colota/locations.db

    partitioning {
        table      locations       # which user table to rotate (repeatable)
        table      _requests       # also rotate the built-in request log
        interval   daily           # daily | weekly | monthly (default daily)
        format     parquet         # parquet | csv (default parquet)
        path       /var/data/colota/archive
        filename   {table}_{date}.parquet   # template: {table}, {date}, {year}, {month}
        keep_days  7               # keep recent N days in active DB before archiving (default 1)
        compress   zstd            # snappy | zstd | gzip | none (default zstd)
        max_age    365d            # auto-delete Parquet files older than this (default: never)
    }
}
```

### Querying across active + archived data

The module automatically rewrites queries that reference a partitioned table to include archived data. In the Caddyfile, you write:

```caddyfile
query get_locations {
    sql `
        SELECT tst, lat, lon, acc, batt, received
        FROM locations
        WHERE ($since IS NULL OR tst > $since)
        ORDER BY tst DESC
        LIMIT $limit
    `
    param since { from query, key since, type timestamp }
    param limit { from query, key limit, type int, default 100, cap 5000 }
    output { format json, envelope on }
}
```

At execution time, the module detects that `locations` is partitioned and rewrites the FROM clause:

```sql
SELECT tst, lat, lon, acc, batt, received
FROM (
    SELECT * FROM locations
    UNION ALL
    SELECT * FROM read_parquet('/var/data/colota/archive/locations_*.parquet')
)
WHERE ($since IS NULL OR tst > $since)
ORDER BY tst DESC
LIMIT $limit
```

This is transparent to the Caddyfile author — no query changes needed.

### Day rollover mechanism

```go
func (m *Middleware) startPartitioner(ctx context.Context) {
    ticker := time.NewTicker(1 * time.Minute)
    for {
        select {
        case <-ticker.C:
            for _, tbl := range m.Partitioning.Tables {
                m.archiveTable(ctx, tbl)
            }
        case <-ctx.Done():
            return
        }
    }
}

func (m *Middleware) archiveTable(ctx context.Context, table string) {
    cutoff := time.Now().AddDate(0, 0, -m.Partitioning.KeepDays)
    dateStr := cutoff.Format("2006-01-02")
    archivePath := filepath.Join(
        m.Partitioning.Path,
        strings.ReplaceAll(m.Partitioning.Filename, "{date}", dateStr),
    )
    archivePath = strings.ReplaceAll(archivePath, "{table}", table)

    // Export rows older than cutoff to Parquet
    exportSQL := fmt.Sprintf(`
        COPY (SELECT * FROM %s WHERE ts < $1)
        TO '%s' (FORMAT PARQUET, COMPRESSION '%s')
    `, table, archivePath, m.Partitioning.Compress)

    _, err := m.db.ExecContext(ctx, exportSQL, cutoff)
    if err != nil {
        m.log.Error("partition export failed", zap.Error(err))
        return
    }

    // Delete exported rows from active DB
    _, err = m.db.ExecContext(ctx, fmt.Sprintf(
        "DELETE FROM %s WHERE ts < $1", table,
    ), cutoff)
    if err != nil {
        m.log.Error("partition cleanup failed", zap.Error(err))
        return
    }

    // Checkpoint to reclaim space
    m.db.ExecContext(ctx, "CHECKPOINT")

    // Purge old Parquet files beyond max_age
    if m.Partitioning.MaxAge > 0 {
        m.purgeOldFiles()
    }
}
```

### Design rationale

| Decision                            | Why                                                                                                   |
| ----------------------------------- | ----------------------------------------------------------------------------------------------------- |
| **Parquet, not separate .db files** | Columnar compression (5-10x smaller); DuckDB reads Parquet natively; no connection pool fragmentation |
| **Keep recent days in active DB**   | Today's queries hit the fast row-store; archive scans are for historical ranges                       |
| **Transparent query rewrite**       | Caddyfile authors never think about partitioning — the module handles it                              |
| **Background goroutine, not cron**  | Keeps everything in-process; no external dependencies                                                 |

## 13. Built-in Request Logging Schema

Even without any `query` blocks, the module logs every request to DuckDB:

```sql
CREATE TABLE IF NOT EXISTS _requests (
    ts           TIMESTAMPTZ NOT NULL,
    ip           VARCHAR     NOT NULL,
    method       VARCHAR(8)  NOT NULL,
    host         VARCHAR     NOT NULL,
    path         VARCHAR     NOT NULL,
    query        VARCHAR,
    status       SMALLINT    NOT NULL,
    latency_ms   INTEGER     NOT NULL,
    bytes_sent   BIGINT      NOT NULL,
    user_agent   VARCHAR
);
```

This table is always available. User-defined queries (via `query` blocks) create **their own tables** — the module does not enforce a schema on user data.

## 14. Provision-Time Validation

When Caddy loads the config (startup or reload), the module:

1. **Opens** a temporary in-memory DuckDB connection
2. **Validates** each query's SQL by replacing `$params` with `NULL` and running `EXPLAIN`
3. **Resolves** `sql_file` paths and reads them
4. **Checks** that all `$param_name` references in SQL have a corresponding `param` block
5. **Verifies** all `route` directives reference existing query names
6. **Fails** the entire config if any validation error occurs (Caddy keeps the old config running)

This means a typo in your SQL or a missing param binding is caught **before** it affects live traffic.

## 16. Implementation Phases

| Phase  | Deliverable                          | Description                                                       |
| ------ | ------------------------------------ | ----------------------------------------------------------------- |
| **1**  | Core module + Caddyfile parser       | `UnmarshalCaddyfile`, `Provision`, `Cleanup`, module registration |
| **2**  | DuckDB connection + schema bootstrap | `db.Open`, built-in `_requests` table, `db_setting` application   |
| **3**  | Async batch writer                   | Channel-based write pipeline for `_requests` logging              |
| **4**  | Query registry + param resolver      | `QueryDef` → prepared statement, `resolveParams` from all sources |
| **5**  | Route dispatch + handler             | `ServeHTTP` → route match → execute query → format output         |
| **6**  | Output formatting                    | JSON (with envelope), CSV, NDJSON, status/body overrides          |
| **7**  | Parquet partitioning + day rollover  | Background archiver, transparent query rewrite, file purging      |
| **8**  | External SQL files                   | `sql_file` loading at provision time                              |
| **9**  | Query result caching                 | In-memory cache with TTL per query                                |
| **10** | Route auth + rate limiting           | `require_header`, per-route `rate_limit`                          |
| **11** | Build + test                         | `build.sh`, integration tests with `caddytest`                    |

### Phase 1–7 = MVP

After phase 7, the Colota GPS use case is fully operational with long-term Parquet archiving. Phases 8–11 are polish.

## 17. Design Decisions

| Decision                                         | Rationale                                                                                |
| ------------------------------------------------ | ---------------------------------------------------------------------------------------- |
| **DuckDB over SQLite**                           | Columnar OLAP engine — aggregations over location history are orders of magnitude faster |
| **Inline SQL in Caddyfile**                      | Zero indirection — the query is right where the route is defined                         |
| **`sql_file` for shared/long queries**           | Keeps Caddyfile readable; enables query reuse across site blocks                         |
| **Provision-time SQL validation**                | Fail-fast on config load, not at request time                                            |
| **Async batch writer for logging**               | Never add latency to the request path for analytics writes                               |
| **Sync execution for user queries**              | The client expects a response — must be synchronous                                      |
| **Named params (`$name`) not positional (`$1`)** | Self-documenting, order-independent, less error-prone                                    |
| **No ORM, no query builder**                     | SQL is the interface — DuckDB's SQL is powerful, no need to abstract it                  |
| **Parquet day-rollover for long-running data**   | Keeps active DB small; Parquet compression shrinks archives 5-10x; DuckDB reads natively |

## 18. Out of Scope for v0.1.0

These are explicitly deferred:

- Abuse detection engine (Tier 1 in-memory + Tier 2 DuckDB deep scans)
- GeoIP lookups
- UA parsing / bot detection
- Scheduled queries (cron)
- WebSocket support
- SurrealDB integration
- SQL file hot-reload via fsnotify
- Materialized stats / hourly aggregations
- Grafana/dashboard integration

These may be addressed in future versions based on need.
